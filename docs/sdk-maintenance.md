# SDK Maintenance Notes

This repo is generated and regenerated frequently. Keep changes deterministic and easy to rebase.

## Commit Strategy

1. Runtime correctness changes in one commit.
2. Lint/trunk scope and config changes in a separate commit.

## Validation Sequence

Run this sequence after runtime edits and after `trunk check --fix --all`:

```sh
./scripts/validate-sdk
```

This executes:

1. `./scripts/test-mock tests/api_resources`
2. `trunk check --fix --all`
3. `./scripts/test-mock tests/api_resources` (again)

## Generated-Surface Policy

- Hand-written core (`_base_client`, transport, auth, shared utils): keep strict.
- Generated `resources/` and `types/`: prefer minimal deterministic shims in shared code over broad generated-file edits.
- Docs/tests generated by codegen: avoid formatting/style churn unless required for correctness.

## Recurring Trunk vs Codegen Conflicts

- `ruff/TC001`, `ruff/TC003`: generated import placement for typing/runtime.
- `ruff/S101`, `bandit/B101`: generated tests rely on `assert`.
- `ruff/INP001`: generated package layout in tests/scripts.
- `ruff/TRY003`, `ruff/EM101`, `ruff/EM102`: generated exception style.
- `pyright/reportMissingModuleSource`: environment/lock-driven optional dependency resolution.
- `checkov`, `trivy` on generated spec/workflow content: high-noise in generated repos.

When these reappear after regeneration, prefer tightening or documenting ignore scope rather than hand-editing generated files.
